# 对话结束记录规范

以后当用户要求“结束对话”“归档”“写 notes”“总结本次对话”或表达类似意图时，在 `notes` 目录中新建一份 Markdown 记录，按本规范执行。

## 文件命名

使用当前 UTC 时间命名：

```txt
notes/YYYY-MM-DDTHH-MM-SSZ.md
```

示例：

```txt
notes/2026-05-01T06-06-02Z.md
```

命名要求：

- 必须使用 UTC 时间，不使用本地时区。
- 使用 `date -u +%Y-%m-%dT%H-%M-%SZ` 获取时间。
- 如果同一秒内已存在同名文件，重新获取当前 UTC 时间。
- 不使用 `session-summary.md` 这类非时间戳名称。

## 内容结构

每份记录应包含以下部分。若某部分确实不适用，可以保留标题并写明“不适用”。

```md
# 本次对话实现记录

## 对话前的目标与状态

## 对话后的最终状态

## 做了哪些事情

## 为什么这么做

## 大致实现过程

## 遇到的问题与解决方式

## 验证结果

## 当前结论
```

## 写作标准

记录要能让之后重新接手的人快速恢复上下文，不只写最终结果，也要写清楚过程中的判断。

必须覆盖：

- 用户最初的目标。
- 用户后续新增或修正的要求。
- 开始前项目的大致状态。
- 结束后项目的大致状态。
- 修改过或重点查看过的文件。
- 核心 API、数据结构或行为变化。
- 关键技术选择，以及为什么这样选。
- 实现的大致步骤。
- 遇到的错误、失败命令或约束。
- 每个错误是如何定位和解决的。
- 最终跑过哪些验证命令。
- 哪些验证通过，哪些没有运行或没有通过。

## 过程描述要求

“大致实现过程”不要只写流水账，应解释每一步背后的关注点。例如：

- 查看 `package.json` 是为了确认脚本、依赖和包管理器。
- 查看 `src/Ability.ts` 是为了确认公开 API 和类型边界。
- 查看 `src/internal/ability.ts` 是为了确认运行时规则匹配和构造逻辑。
- 查看测试文件是为了确认行为约束是否被覆盖。
- 搜索旧 API 是为了避免文档、测试或示例残留过期写法。

描述技术选择时，要写清楚取舍。例如：

- 为什么某个函数应该是同步的。
- 为什么错误应该进入 Effect error channel。
- 为什么保留或删除某个 API。
- 为什么使用某个测试方式。
- 为什么某个实现更贴合当前代码组织。

## 错误记录要求

遇到错误时，不要只写“已解决”。应包含：

- 出错命令或操作。
- 关键错误信息。
- 判断它为什么发生。
- 采取了什么解决方式。
- 解决后如何验证。

示例：

````md
运行 `yarn tsx examples/basic.ts` 时失败：

```txt
Error: listen EPERM: operation not permitted /tmp/tsx-1000/38.pipe
```

判断原因是 `tsx` 需要在 `/tmp` 下创建 IPC pipe，但当前沙箱不允许。随后使用提升权限重新运行示例，命令通过，并确认输出符合预期。
````

## 验证记录要求

记录验证结果时，优先写实际执行过的命令：

```sh
yarn typecheck
yarn test
yarn tsx examples/basic.ts
git diff --check
```

如果某个验证没有运行，要说明原因。不要暗示未执行的验证已经通过。

## 语言与风格

- 使用中文。
- 保持事实性和可追溯性。
- 不写空泛评价。
- 不夸大结果。
- 不把猜测写成事实；如果是推断，明确写“推断”。
- 记录应足够具体，但不需要复制完整 diff。
- 文件路径用反引号标记，例如 `src/Ability.ts`。

## 完成后的检查

写完记录后执行：

```sh
find notes -maxdepth 1 -type f | sort
git diff --check
```

如果本次只是新增 notes 文档，一般不需要运行项目测试。若 notes 中记录了代码改动已经验证过，应以实际执行结果为准。
