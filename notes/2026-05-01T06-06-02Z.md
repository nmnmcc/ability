# 本次对话实现记录

## 对话前的目标与状态

本次对话的目标是实现一个受 Effect 与 CASL 启发的 TypeScript 授权库。核心诉求不是简单做一个权限判断函数，而是让 API、代码组织和错误模型尽量贴近 Effect 的风格：

- 使用模块化函数 API，例如 `Ability.define()`、`Ability.check()`。
- 使用生成器 DSL 定义规则，让规则定义自然接近 Effect 的 `Effect.gen` 写法。
- 从 `Ability.define` 里 yield 出来的规则自动推导 Ability 类型。
- action、subject、field、subject value 都要从规则中获得类型约束。
- predicate 可以是同步布尔值，也可以是 `Effect<boolean, E, R>`，并且错误 `E` 与服务依赖 `R` 要体现在 `Ability.check` 的类型里。
- 新增 `examples` 和 `tests` 文件夹，并使用 `@effect/vitest` 编写测试。

过程中有几次重要的需求调整：

- `Ability.define` 最初可以按 Effect 风格返回 `Effect<Ability>`，但后来明确改为直接返回 `Ability`，因为定义 Ability 本身没有副作用。
- 曾经拆分过 `Ability.check` 和 `Ability.allows`，之后又按要求合并为一个 API。
- 曾经把 `Ability.check` 改成返回 `Effect<Result<...>>`，最后又按要求改回 Effect 失败通道语义，并且只保留 `Ability.check`。

## 对话后的最终状态

最终公开 API 形态如下：

```ts
const ability = Ability.define<Subjects>()(function* (ability) {
  yield* ability.allow("read", "Post")
  yield* ability.deny("delete", "Post", {
    when: (post) => post.published,
    reason: "Published posts cannot be deleted"
  })
})

yield* Ability.check(ability, {
  action: "read",
  subject: "Post",
  value: post
})
```

最终语义是：

- `Ability.define` 是纯同步构造器，直接返回不可变的 `Ability`。
- `ability.allow` / `ability.deny` 直接返回 `Rule`，但 `Rule` 实现了 `Effect.Yieldable` 与迭代协议，所以可以 `yield* ability.allow(...)`。
- `Ability.check` 是唯一的权限检查入口。
- `Ability.check` 成功时返回 `void`。
- 没有规则允许请求时，`Ability.check` 通过 Effect error channel 失败为 `AuthorizationError`。
- predicate 返回的 `Effect` 如果失败，其错误会保留在 `Ability.check` 的错误类型中。
- `Ability.allows` 已移除。
- `Result` 语义已移除。

最终涉及的主要文件：

- `src/Ability.ts`：公开类型与公开 API。
- `src/internal/ability.ts`：内部实现。
- `src/index.ts`：模块导出。
- `examples/basic.ts`：可运行示例。
- `tests/Ability.test.ts`：运行时测试，使用 `@effect/vitest`。
- `tests/Ability.type-tests.ts`：类型层面的约束测试。
- `README.md`：使用说明。
- `vitest.config.ts`：测试入口配置。
- `package.json`：脚本与依赖。

## 做了哪些事情

1. 建立 Ability 的类型模型

   在 `src/Ability.ts` 中定义了 `SubjectMap`、`SubjectName`、`FieldOf`、`Rule`、`Ability`、`Builder`、`CheckRequest` 等类型。这样做的目的是让规则本身成为类型来源，而不是要求用户额外维护一套 action/subject 权限枚举。

2. 实现生成器 DSL

   在 `src/internal/ability.ts` 中实现了 `RuleProto`，让 `Rule` 可以被 `yield*`。这样 `Ability.define<Subjects>()(function* (ability) { ... })` 看起来像 Effect 的生成器 DSL，但定义过程仍然是同步、纯净的。

3. 实现不可变规则与 Ability

   `makeRule`、`freezeRule` 和 `make` 会冻结规则数组、字段数组和 Ability 实例中的规则引用。这样做是为了让 Ability 定义完成后保持稳定，避免运行时被外部代码修改。

4. 实现规则匹配

   内部通过 `matchesField`、`matchesCondition`、`matchesRule` 和 `findRule` 完成匹配。规则从后往前查找，保留 CASL 中常见的“后定义规则优先”语义。

5. 实现类型化错误

   `AuthorizationError` 使用 `Data.TaggedError` 定义。predicate 可以返回 `Effect<boolean, E, R>`，`Ability.check` 的类型会根据匹配规则推导出 `AuthorizationError | E` 与所需服务 `R`。

6. 增加测试与示例

   新增了 `examples/basic.ts` 和 `tests` 目录。测试覆盖了同步定义、允许规则、默认拒绝、后定义规则优先、字段匹配和 predicate 错误传播。

7. 按用户反馈调整 API

   最后将 API 收敛为：

   - `Ability.define`：同步返回 `Ability`。
   - `Ability.check`：返回 `Effect<void, AuthorizationError | PredicateError, R>`。
   - 不提供 `Ability.allows`。
   - 不使用 `Result` 作为权限检查结果。

## 为什么这么做

`Ability.define` 不返回 Effect 的原因是：定义规则只是构造数据，没有 I/O、没有环境依赖、没有异步边界，也不需要失败通道。把它包成 Effect 会让用户在创建 Ability 时多写一次 `yield*`，但不会带来真实收益。

`Rule` 仍然支持 `yield*` 的原因是：用户想要 Effect 风格的生成器定义体验。把 `Rule` 做成 `Yieldable` 可以同时满足这两个目标：

- 定义过程是同步纯函数。
- 语法仍然接近 `Effect.gen`。

`Ability.check` 使用 Effect 失败通道而不是 `Result` 的原因是：predicate 本身可能是 Effect。如果 check 返回 `Result`，predicate 的失败必须被塞进成功通道里的 `Result.Failure`，这会削弱 Effect 的错误模型。回到失败通道后，授权失败和 predicate 失败都能被 `Effect.catchTag`、`Effect.catchAll` 等 Effect 组合子自然处理。

只保留 `Ability.check` 的原因是：`allows` 和 `check` 容易造成语义分裂。现在需要布尔值时，可以用 Effect 组合子派生：

```ts
const allowed = yield* Ability.check(ability, request).pipe(
  Effect.match({
    onFailure: () => false,
    onSuccess: () => true
  })
)
```

这样核心 API 更小，错误语义也更一致。

然后围绕几个问题做选择：

- 公共 API 是否应该是类：最终选择模块函数，与 Effect 风格保持一致。
- `define` 是否应该返回 Effect：最终选择同步返回 Ability，因为没有副作用。
- builder 的 `allow` / `deny` 是否应该返回 Effect：最终选择返回 Rule，让 Rule 自身实现 Yieldable。
- 检查接口是否需要两个函数：最终只保留 `Ability.check`。
- 拒绝结果放在哪里：最终选择 Effect error channel，而不是 `Result`。
- 规则优先级如何处理：选择后定义规则优先，从规则数组尾部向前匹配。
- 字段约束如何表达：`CheckRequest` 根据匹配 rule 的 `fields` 推导允许字段。
- predicate 错误如何保留：通过 `RuleError<MatchingRules<...>>` 从规则类型中抽取错误。

8. 跑类型检查、测试和示例验证结果。

## 遇到的问题与解决方式

1. `Ability.define` 一开始过度 Effect 化

   问题：把定义 Ability 做成 `Effect<Ability>` 虽然形式上很 Effect，但定义规则本身没有副作用。

   解决：将 `Ability.define` 改成同步函数；让 `Rule` 实现 `Effect.Yieldable`，保留生成器 DSL 的使用体验。

2. `Ability.check` 与 `Ability.allows` 语义重复

   问题：两个入口会让用户在“想要错误”与“想要布尔值”之间做 API 选择，同时增加维护面。

   解决：移除 `Ability.allows`，只保留 `Ability.check`。需要布尔值时由调用方用 `Effect.match` 派生。

3. `Result` 版本不符合最终错误模型

   问题：`Effect<Result<void, Error>, never, R>` 会把授权失败放入成功通道，predicate 失败也会被转换为 `Result.Failure`。这和 Effect 的类型化失败通道不够一致。

   解决：移除 `Result` import 和相关类型，把 `Ability.check` 改回 `Effect<void, AuthorizationError | PredicateError, R>`。

4. 示例运行时遇到沙箱错误

   运行 `yarn tsx examples/basic.ts` 时，tsx 需要在 `/tmp/tsx-1000/*.pipe` 创建 IPC pipe，沙箱中第一次运行失败：

   ```txt
   Error: listen EPERM: operation not permitted /tmp/tsx-1000/38.pipe
   ```

   解决：使用提升权限重新运行示例，命令成功，输出为：

   ```ts
   {
     canUpdateTitle: true,
     deleteResult: "Published posts cannot be deleted"
   }
   ```

5. 需要防止旧 API 残留

   问题：在多次 API 调整后，容易在 README、测试或示例中残留 `Result` 或 `Ability.allows`。

   解决：使用 `rg "Result|allows"` 和 `rg "Ability\\.allows|export const allows|allows:"` 搜索确认残留，随后再次运行类型检查和测试。

当前实现已经满足本次对话最终确定的 API 方向：
