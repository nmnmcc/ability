## 为什么这么做

`references/effect-smol` 的核心风格之一是模块函数加不可变 runtime value，并让公开值参与 Effect 生态协议，例如 `Pipeable`、`Equal`、`Hash` 和 `Inspectable`。原始实现已经使用了 Effect API 和 generator-friendly rule，但协议层、错误建模和 dual API 还不完整。

保留 `Ability.define()` 同步，是因为这是仓库明确的 API invariant，也符合当前规则收集模型：`yield* ability.allow(...)` 只用于同步收集 `Rule`，不应该把定义阶段变成 Effect。

将 predicate 同步 throw 转换为 `PredicateEvaluationError`，是为了满足“predicate failures should use the Effect error channel”的约束；但 predicate 返回 Effect 时不包裹其失败，是为了保留用户 predicate 的原始错误类型和 service 类型。

`AbilityRef` 是包内显式 mutable boundary。它的 equality 使用 referential identity，而不是 current ability 的结构相等，因为引用当前值会变化；这比把 mutable 状态参与结构 equality 更可预测。

`AnyRule` 中的 `any` 保留，是因为它位于公共宽类型边界。改成 `unknown` 或 `never` 会让现有泛型约束和规则推断大面积退化，风险超过收益。

## 大致实现过程

先阅读并对齐仓库规则，确认不能修改 `references/effect-smol/`，也不能改变 `Ability.define`、`Ability.check` 等公共 API 语义。随后围绕 `src/Ability.ts` 和 `src/internal/ability.ts` 检查公开类型和 runtime 构造逻辑，确定需要补齐的是协议支持、错误建模和 guard，而不是重写授权匹配算法。

接着在 `src/internal/ability.ts` 中补齐 runtime prototypes。这里保持原有 last matching rule wins 的行为，仍从后往前匹配规则，只在值协议、冻结、错误通道和 guard 上做增强。

然后处理 `AbilityExtra` 和 `AbilityRef`。这两个模块的主要目标是更贴近 Effect module function 风格，因此把核心实现拆成 effectful 内部函数，再用 `dual` 暴露 data-first / data-last 两种调用形式。这样用户可以继续直接调用，也可以在 `pipe` 中组合。

之后补测试。行为测试覆盖新增 runtime 行为，类型测试覆盖错误类型传播，避免只在运行时通过但 public API 类型退化。

最后执行验证命令，并做静态扫描检查：确认 `src` 中没有 `Effect.gen`、`async`、`await`、手写 `try/catch` 命中；确认测试没有 `Effect.runSync` 和 `expect(` 命中；确认 `src` 没有 tab 或行首分号残留。

## 遇到的问题与解决方式

曾尝试将公共 `AnyRule` 中的 `any` 改成更窄类型。结果导致大量类型错误，说明它当前承担公共宽类型边界职责。最终回退该尝试，保留 `AnyRule` 的 `any`，只清理内部可控位置。

运行 `yarn tsx examples/basic.ts` 时在普通沙箱下失败：

```txt
Error: listen EPERM: operation not permitted /tmp/tsx-1000/38.pipe
```

判断原因是 `tsx` 需要在 `/tmp` 下创建 IPC pipe，但当前沙箱限制该操作。随后按仓库说明使用提升权限重新运行 `yarn tsx examples/basic.ts`，命令通过，并确认输出符合预期。

在清理内部 `any` 时，曾把 `validateRawConditions` 参数收窄为 `Ability.MongoCondition<object>`，导致 `yarn typecheck` 失败。关键错误是 `MongoCondition<SubjectUnion<Subjects>>` 不能赋给 `MongoCondition<object>`，原因是 `$where` 的 `this` 类型使该类型不是协变的。最终将 `validateRawConditions` 改为泛型函数，并只在调用 `@ucast/mongo2js` 的 `guard` 边界使用 `unknown` 到 `MongoQuery<Record<string, unknown>>` 的转换。修复后 `yarn typecheck` 通过。

复查 `AbilityRef.set` 时发现一处为避开 ASI 的行首分号。功能正确，但和目标风格不一致，随后改成显式 `mutable` 绑定，消除行首分号。
