## 对话前的目标与状态

用户最初目标是把项目改成“官方 CASL 内核 + 本项目 Effect 适配层”，希望一键达到生产可用状态，同时基本保留当前 API 和类型安全开发体验。

开始前项目是一个 TypeScript 授权库，主要入口是 `src/index.ts` 和 `src/Ability.ts`，内部实现集中在 `src/internal/ability.ts`。项目已有 Effect 风格 API：`Ability.define()` 同步返回 `Ability`，`ability.allow()` / `ability.deny()` 返回可 `yield*` 的 `Rule`，`Ability.check()` 返回 `Effect<void, ...>`。当时仍存在与 CASL 行为不一致的功能和 API，例如 `when` predicate、`AbilityRef`、非 CASL 原生条件模型等。

用户后续要求先列出与 CASL 行为不同的地方，然后指定实现计划：

- 删除 `when` 功能，对齐 CASL。
- 废除 `AbilityRef`，改为 `Ability.update` 返回全新的 `Ability`。
- 允许配置 `anyAction` / `anySubjectType` 的计划后来被用户撤回，因此仍保留固定 `"manage"` / `"all"`。
- 不要更宽的类型，条件类型要更严格地贴近 CASL。

最后用户指出 `examples/acbc.ts` 如果太复杂会影响使用意愿，要求参考 `references` 中 Effect 最佳实践，并进一步要求用 `Effect.gen` 包裹 `Ability.define` 来重构示例。

## 为什么这么做

删除 `when` 是为了对齐 CASL 的核心行为。CASL 的规则条件是可序列化、可传输、可转换查询的结构化条件；任意 predicate 虽然灵活，但会破坏 CASL 兼容层的可预测性，也让 `toRawRules` 和 `toMongoAbility` 出现无法序列化的分支。

废除 `AbilityRef` 是为了保留 immutable ability 模型。用户明确要求 update 返回全新 Ability，因此用 `Ability.update(self, rawRules)` 或 data-last 形式替代可变引用。

条件类型改成 CASL `MongoQuery` 是为了避免本项目定义一个比 CASL 更宽的条件 DSL。这样能更早暴露不被 CASL 支持或类型不稳定的条件写法。

没有实现 `anyAction` / `anySubjectType` 配置，是因为用户后续撤回了该计划。本次仍固定 CASL 默认语义里的 `"manage"` 和 `"all"`。

`examples/acbc.ts` 最终采用 `Effect.gen` 包裹 `Ability.define`，是为了表达 Effect 的惰性环境读取：`Session` 作为请求环境服务在 `Effect.gen` 中读取，读取后同步构造 `Ability`。这样没有改变核心 API 约束：`Ability.define()` 本身仍是纯同步构造器并直接返回 `Ability`。

## 大致实现过程

先阅读 `package.json`、`src/Ability.ts`、`src/internal/ability.ts`、测试和示例，确认公开 API、内部匹配逻辑和现有行为约束。随后添加 CASL 依赖并把 runtime 条件匹配迁移到 CASL 语义，重点保持“最后匹配规则胜出”的扫描方式。

然后删除 predicate / `when` 路径，包括公开类型、内部 runtime rule 字段、predicate error、序列化错误和相关测试。`Ability.check` 仍然保留 Effect error channel：授权失败走 `AuthorizationError`，条件匹配失败走 condition 相关错误。

接着删除 `AbilityRef`，清理 public export、package subpath export、构建入口、示例和测试。用 `Ability.update` 接收 raw rules 并复用原 ability options，返回新的 ability。

之后更新示例和 README，确保用户面对的是 CASL condition 模型而不是旧 predicate 模型。`examples/advanced.ts` 改为展示 `Ability.update` 的 immutable 更新方式。

用户指出 `examples/acbc.ts` 太复杂后，先读取 `references/effect-smol/LLMS.md` 和 `references/effect-smol/AGENTS.md`，确认 Effect references 推荐 `Effect.gen` / `Effect.fn` 风格，`Context.Service` 用于真实环境依赖。最初把示例简化为普通 session 常量，后来用户明确要求“直接 `Effect.gen` 包裹 `Ability.define`”，于是再次重构为：

- `class CurrentSession extends Context.Service<CurrentSession, Session>()("CurrentSession") {}`
- `const makeAbility = Effect.gen(function* () { const session = yield* CurrentSession; return Ability.define<Subjects>()(...) })`
- `program` 中先 `yield* makeAbility`，再执行 `Ability.check`。
- 末尾用 `Effect.provideService(CurrentSession, {...})` 注入示例 session。

## 遇到的问题与解决方式

运行示例时普通 sandbox 下 `tsx` 失败：

```txt
Error: listen EPERM: operation not permitted /tmp/tsx-1000/38.pipe
```

判断原因是 `tsx` 需要在 `/tmp` 下创建 IPC pipe，但当前沙箱不允许。随后按仓库说明使用提升权限重新运行 `yarn tsx examples/...`，示例命令通过，并确认输出符合预期。

`examples/acbc.ts` 的设计经历了一次方向调整。最初为了展示 ABAC/CBAC 和 Effect 环境，把示例写得过重，包含多套 context、嵌套分支和泛型 helper。用户认为这会影响库的采用。随后先简化到普通 session 数据，再根据用户进一步要求，恢复 `Context.Service`，但把读取 session 放在外层 `Effect.gen`，避免把 `yield* Session` 直接放进 `Ability.define` 的 rule DSL 中。

工作树中存在 unrelated dirty changes。遵守仓库规则，没有使用 `git reset` 或 `git checkout --` 回滚，也没有修改 `references/effect-smol/`。
