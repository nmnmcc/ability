## 对话前的目标与状态

用户最初询问该项目能否改为 CASL 的兼容层，以一键达到生产可用状态。随后用户明确修正目标：不是自研 CASL 复刻，而是采用“官方 CASL 内核 + 本项目 Effect 适配层”，并且基本保留当前 API 以及类型安全的开发体验。

## 对话后的最终状态

项目已经改为使用官方 `@casl/ability@^6.8.1` 作为规则索引、action/subject/field 匹配和 Mongo-style 条件匹配内核。当前 Effect 风格 API 仍然保留：

- `Ability.define` 仍是同步 generator DSL。
- `Ability.check` 仍返回 `Effect<void, ...>`。
- `when` 谓词仍支持同步 boolean 或 `Effect<boolean, E, R>`。
- predicate throw、Effect fail、service requirement 仍走 Effect error channel。
- last matching rule wins 的行为通过 CASL candidate rules + 本项目 predicate 层继续保持。

新增了 `src/Casl.ts`，作为 CASL 兼容模块：

- 重导出官方 `@casl/ability` 的 `AbilityBuilder`、`createMongoAbility`、`defineAbility`、`PureAbility`、`ForbiddenError`、`subject` 等 API。

- `Casl.toMongoAbility` 默认 strict，遇到带 `when` 的规则会失败为 `SerializationError`，避免静默丢弃 Effect predicate。

## 为什么这么做

用户的目标是生产可用，而不是维护一个长期追赶 CASL 行为的自研实现。官方 CASL 已经提供成熟的 rule index、Mongo 条件 matcher、field matcher、action resolver、`can/cannot` API 和生态集成。把 CASL 放到内核层能减少本项目维护面，并让未来接入 `@casl/mongoose`、`@casl/prisma`、React/Vue 等官方生态更直接。

同时，当前项目的价值不只是 CASL 规则模型，还包括 Effect 风格 API、类型推导、Effect error channel 和 Effectful predicates。CASL 的同步 `can()` 无法无损表达 `Effect<boolean, E, R>` 谓词，所以实现没有把当前 API 替换成 CASL API，而是把 CASL 作为候选规则和条件匹配内核，再在候选规则命中后执行本项目的 `when` 层。

`Casl.toMongoAbility` 默认 strict 是为了避免安全风险：带 `when` 的规则如果直接导出为 CASL raw rules，CASL 无法表示这些函数，静默忽略会造成授权结果变宽或变窄。因此默认失败，调用者必须显式传 `{ strict: false }` 才能导出忽略 `when` 的 CASL ability。

`MongoCondition` 类型没有直接复用 CASL 的 `MongoQuery` 类型，是因为接入后发现 CASL 当前类型对数组字段 `$in` 的推导过窄，现有测试和示例中的 `tags: { $in: ["effect"] }` 无法通过类型检查。最终保留本项目自己的条件类型，既给常见 operators 提供约束，也允许 dot-path 和复杂 Mongo-style 查询继续存在。

## 大致实现过程

先查看 `package.json`、`tsconfig.json`、`tsdown.config.ts`，确认依赖、脚本、打包入口和 TypeScript 严格选项。随后查看 `src/Ability.ts`、`src/internal/ability.ts`、`tests/Ability.test.ts` 和 `README.md`，确认当前公开 API、运行时规则结构和测试覆盖。

然后查阅 CASL 官方资料和安装 `@casl/ability@^6.8.1`。安装后通过 Node 运行时和 `node_modules/@casl/ability/dist/types/*.d.ts` 确认官方导出：`createMongoAbility`、`AbilityBuilder`、`PureAbility`、`ForbiddenError`、`subject`、`possibleRulesFor`、`rulesFor`、`actionsFor`、`Rule.matchesConditions`、`Rule.matchesField` 等。

接着在 `src/internal/ability.ts` 中新增隐藏的 `CaslState`。构造 Ability 时仍冻结本项目的 `Rule` 值，但同时把这些规则转换为 CASL raw rules，构造官方 `createMongoAbility` 实例。通过 `WeakMap` 将 CASL raw rule 映射回本项目的 frozen runtime rule，这样公开返回值仍是本项目的 `Rule`，不会泄漏 CASL Rule 对象。

匹配流程改为：

1. 将本项目 check request 规范化为 runtime request。
2. 调用 CASL `possibleRulesFor(action, subject)` 获得按 CASL 优先级排序的候选规则。
3. 用 CASL `matchesField` 和 `matchesConditions` 处理字段与 Mongo 条件。
4. 用本项目 `matchesPredicate` 处理 `when`，保留 Effect 类型、错误和服务需求。
5. 找到第一个匹配候选后，如果 `_tag === "Allow"` 则 `Ability.check` 成功，否则失败为 `AuthorizationError`。

然后新增 `src/Casl.ts`，重导出官方 API，并实现 `toMongoAbility`。该适配器通过 `Ability.toRawRules` 导出 raw rules，继承本项目 action aliases 和 subject detection 逻辑，再构造官方 CASL Mongo ability。

最后更新文档、测试和类型测试，跑完整验证。

## 遇到的问题与解决方式

运行 `yarn typecheck` 时首先失败，关键错误是现有示例和测试中的数组字段 `$in` 条件不满足 CASL `MongoQuery` 类型：

```txt
Type '{ $in: string[]; }' is not assignable to type ...
Type 'string[]' is not assignable to type '(readonly string[])[]'
```

判断原因是 CASL 的 `MongoQuery` 泛型对数组字段的 `$in` 推导与本项目既有用法不兼容。解决方式是取消 `src/Ability.ts` 对 CASL `MongoQuery` 类型的直接导入，改成本项目自己的 `MongoCondition` / `MongoConditionValue` 类型。这样保留已知字段 operator 提示，同时继续允许 dot-path 和扩展查询。修改后 `yarn typecheck` 通过。

接入 CASL 后还遇到 `matchesConditions` 的类型参数问题：

```txt
Argument of type 'object | undefined' is not assignable to parameter of type 'string | undefined'
```

原因是最初将 internal CASL ability 类型写成 `MongoAbility<[string, string], any>`，CASL 认为 subject 参数只能是字符串。解决方式是将运行时 CASL ability 类型调整为 `MongoAbility<[string, string | object], any>`，允许对 subject object 执行条件匹配。

`src/Casl.ts` 中 `copySingleOrArray` 也出现 readonly array 到 mutable array 的类型错误。解决方式是对 readonly array 显式 `Array.from(value as ReadonlyArray<A>)`，返回 CASL raw rule 需要的 mutable array。

运行 `yarn tsx examples/basic.ts` 时第一次失败：

```txt
Error: listen EPERM: operation not permitted /tmp/tsx-1000/38.pipe
```

判断原因是 `tsx` 需要在 `/tmp` 下创建 IPC pipe，但当前沙箱不允许。按 `AGENTS.md` 的测试说明，使用提升权限重新运行 `yarn tsx examples/basic.ts`，命令通过，输出为：

```txt
{
  canUpdateTitle: true,
  deleteResult: 'Published posts cannot be deleted'
}
```
