## 为什么这么做

不添加同步 API 的原因是用户明确要求这是 Effect 系统的库，并且当前库支持 `Effect<boolean, E, R>` predicate。同步 `can/cannot` 无法在保留 service 依赖和错误通道的同时保持可靠语义，所以仍然只提供 `Ability.check`，需要 boolean 时用 `Effect.match` 派生。

排除数据库能力的原因是用户明确要求“数据库方面先不动”。因此本轮只使用 `@ucast/mongo2js` 做内存对象匹配，不导出 AST、query，也不实现 ORM adapter。

使用 `@ucast/mongo2js` 的原因是 CASL 本身也基于 ucast 生态实现 Mongo-style conditions。这样可以避免自研条件语言带来的兼容和维护成本。

`Ability.subject` 采用 wrapper 而不是 CASL 的 DTO mutation，是为了更符合函数式哲学：不修改用户传入对象，不添加隐藏属性。predicate 和 conditions 内部会 unwrap，用户业务对象保持原样。

`AbilityRef` 独立于 `Ability` 的原因是保持 `Ability` immutable。需要动态更新的应用可以显式使用 mutable boundary，但核心授权模型仍是纯数据。

`permittedFields` 返回 Effect 的原因是字段计算可能需要评估 Effect predicate。如果做同步 helper，会破坏 predicate 的错误和 service 类型传播。

## 遇到的问题与解决方式

1. `tsx` 示例运行遇到沙箱 IPC 错误

   运行 `yarn tsx examples/basic.ts` 和 `yarn tsx examples/advanced.ts` 时，沙箱中失败：

   ```txt
   Error: listen EPERM: operation not permitted /tmp/tsx-1000/38.pipe
   ```

   判断原因是 `tsx` 需要在 `/tmp` 下创建 IPC pipe，而当前沙箱限制了该操作。随后按仓库说明使用提升权限重新运行两个示例，均通过。

2. `AbilityRef` 类型过窄

   初版 `AbilityRef` 保留了精确 `Ability<Subjects, Rules>` rule union，导致 `AbilityRef.set(ref, nextAbility)` 在 next rule union 不同的时候无法通过类型检查。

   解决方式是把 `AbilityRef<Subjects>` 设计为持有 `Ability<Subjects, AnyRule>`。这符合动态容器语义，也避免把当前规则 union 固定死。

3. `permittedFields` 合并顺序错误

   初版按从后往前处理规则，导致后置 deny 删除字段后，前面的 allow 又把字段加回来。

   解决方式是按规则定义顺序应用字段集合变更：allow 添加字段，deny 删除字段。这样后定义规则可以覆盖前面规则。

4. Condition 类型和 ucast 类型边界较复杂

   `@ucast/mongo2js` 的 `guard` 类型对泛型对象较严格，项目启用了 `exactOptionalPropertyTypes` 和 `noUncheckedIndexedAccess`，初版有多处类型不兼容。

   解决方式是在 `MongoCondition` public 类型上保留 `MongoQuery` 约束，同时在调用 `guard` 的内部边界使用窄范围 `as any`。该 cast 被限制在外部库调用边界，没有扩散到 public API。

5. Effect error channel 让示例返回类型更宽

   Advanced 示例中 `AbilityRef.update` 和 `Ability.check` 可能产生 Effect 错误，直接 `Effect.runSync(program)` 要求环境和错误被处理。

   解决方式是在示例中显式处理 `RawRuleError`、`AuthorizationError` 以及其他错误路径，让示例程序可同步运行。
