## 为什么这么做

用户要求先尝试通过依赖注入实现 ACBC。检查现有代码后判断可以直接实现，原因是：

- `Predicate<Subject, E, R>` 已允许返回 `Effect<boolean, E, R>`。
- `Ability.check` 的公开类型会从匹配规则中提取 `RuleError` 和 `RuleServices`。
- `matchesPredicate` 在运行时会识别 predicate 返回值是否为 Effect，是 Effect 时直接 yield，使服务依赖由外部 Effect 环境提供。

因此不需要把 `Ability.define` 改成 effectful，也不需要改成 mutable class API。把上下文依赖放在 predicate 返回的 `Effect` 里，符合仓库约束：Ability 构造仍是纯同步的，授权检查仍是唯一 effectful 边界。

示例里同时注入 `CurrentAbility`，是为了展示业务侧也可以把当前 Ability 当成 Effect service，而不是在每个业务函数参数中显式传递。这保持了 `Ability` immutable，同时让应用层决定如何提供当前策略。

## 遇到的问题与解决方式

运行新示例时，首次命令失败：

```sh
yarn tsx examples/acbc.ts
```

关键错误信息：

```txt
Error: listen EPERM: operation not permitted /tmp/tsx-1000/38.pipe
```

判断原因是 `tsx` 需要在 `/tmp` 下创建 IPC pipe，但当前沙箱不允许。这个失败与仓库 `AGENTS.md` 中的说明一致：`tsx` 可能需要在 `/tmp` 创建 IPC pipe，沙箱执行失败时应使用适当权限重跑。

随后使用提升权限重新运行：

```sh
yarn tsx examples/acbc.ts
```

重跑通过，输出显示：

```txt
readOwnRestricted: { allowed: true }
updateBody: { allowed: true }
publishAfterHours: { allowed: false, reason: 'AuthorizationError: Publishing requires a trusted network during business hours' }
crossTenantRead: { allowed: false, reason: 'AuthorizationError' }
```

另一个小问题是 `git diff -- examples/acbc.ts` 对未跟踪文件无输出。随后使用 `sed -n '1,260p' examples/acbc.ts` 查看文件内容，并用 `git diff --no-index /dev/null examples/acbc.ts` 检查新增文件 diff。
