2. 实现 action alias

   在 `src/internal/ability.ts` 中实现 alias 校验和展开：

   - alias key 不能是 `manage`。
   - alias target 不能是 `manage`。
   - alias 不能空数组。
   - 检测自引用和循环引用。
   - alias 是单向的：`modify -> update/delete` 时，`modify` 规则能匹配 `update` 和 `delete`，但单独的 `update/delete` 规则不自动匹配 `modify`。

3. 实现内部规则索引

   在 `Ability.make` 时构建隐藏规则索引，按 subject/action 保存规则，构建时展开 action 数组、subject 数组和 alias。索引用于：

   - `Ability.check`
   - `Ability.permittedFields`
   - `possibleRulesFor`
   - `rulesFor`
   - `relevantRuleFor`
   - `actionsFor`

   `check` 和 `relevantRuleFor` 使用高优先级优先顺序，保留后定义规则优先；`permittedFields` 继续按定义顺序应用 allow/deny 字段变化，保持后定义规则覆盖前面规则的行为。

4. 新增 `AbilityExtra`

   新增 `src/AbilityExtra.ts` 并在 `src/index.ts`、`package.json` 中导出：

   - `packRules`
   - `unpackRules`
   - `rulesToFields`
   - `rulesToCondition`
   - `rulesToQuery`

   `packRules`/`unpackRules` 使用 tuple 形式压缩 raw rules，但不使用逗号 join，避免 action/field 名称里有逗号时丢信息。

   `rulesToFields` 提取 allow rule 中的 scalar condition 值，支持 dot path，并跳过 `__proto__`、`constructor`、`prototype` 等危险路径。

   `rulesToCondition` 使用调用方提供的 `and`、`or`、`not`、`empty` hooks 生成 ORM 无关逻辑条件；遇到 `when` predicate 时返回 `QueryGenerationError`。

   `rulesToQuery` 是 `rulesToCondition` 的通用逻辑对象包装，不声明可直接用于 Mongo/Prisma/SQL。

## 为什么这么做

不新增同步 API 的原因是 `notes/` 中已经确认本库核心价值是自动类型安全和 Effect 错误/服务类型传播。同步 `can/cannot` 无法表达 Effect predicate 的错误与依赖，因此继续只保留 `Ability.check` 作为授权检查入口。

alias 放在 `AbilityOptions.actionAliases` 而不是修改 rule 数据，是为了保持 RawRule 和 Rule 的可读性与序列化稳定性。`toRawRules` 输出原始 action，不把 alias 展开后的 action 写回 raw rule。

规则索引作为内部隐藏结构而不是 public interface，是为了不扩大 Ability 的公开数据模型。用户仍只能看到 immutable rules 和 options，运行时可以用索引优化候选规则选择。

`possibleRulesFor`、`rulesFor`、`relevantRuleFor`、`actionsFor` 都返回 Effect，是因为 request 可能只提供 `value`，subject detection 可能失败；`relevantRuleFor` 还可能评估 condition 和 Effect predicate。

`AbilityExtra` 独立成模块，是为了把核心授权检查和派生工具分开。核心模块保留最小授权语义，Extra 提供缓存、调试、查询生成基础。

`rulesToCondition` 遇到 `when` 失败，是因为函数 predicate 不能安全转为数据条件。如果静默忽略，可能生成过宽或过窄的查询，安全性不可控。

继续使用纯 wrapper 版 `Ability.subject`，是因为之前 notes 已确认该设计更符合函数式约束，不修改用户 DTO。

## 遇到的问题与解决方式

1. `AbilityExtra.rulesToFields` 中规则类型过窄

   初次 `yarn typecheck` 报错：

   ```txt
   Property '_tag' does not exist on type 'SubjectMatches<ActionMatches<...>>'
   Property 'conditions' does not exist on type 'SubjectMatches<ActionMatches<...>>'
   ```

   原因是 `Ability.rulesFor` 的返回类型是按 request 匹配后的条件类型，TypeScript 在 Extra 的泛型上下文里不能证明它一定是 `AnyRule` 结构。解决方式是在 Extra 内部把结果收窄为 `ReadonlyArray<Ability.AnyRule>`，这是内部实现边界，public API 仍保持类型化。

2. `rulesToQuery` 的 hooks 返回类型不匹配

   初次 `yarn typecheck` 报错：

   ```txt
   Type '{ $and: readonly Query[]; }' is not assignable to type 'Query'
   Type '{}' is not assignable to type 'Query'
   ```

   原因是 `rulesToCondition` 的 hooks 统一返回 `Query`，而 `rulesToQuery` 需要返回 `LogicalQuery<Query>`。解决方式是让 `rulesToQuery` 内部把 `convert` 结果提升为 `LogicalQuery<Query>`，并让 hooks 返回 `LogicalQuery<Query>`。

3. hidden index 属性和 symbol index type 冲突

   初次 `yarn typecheck` 报错：

   ```txt
   Property '[RuleIndexTypeId]' of type 'RuleIndex' is not assignable to 'symbol' index type 'symbol'
   ```

   原因是 `Ability.Ability` 继承 `Pipeable.Pipeable` 后，直接在 `RuntimeAbility interface` 上声明 unique symbol 属性会和 Pipeable 的 symbol 索引发生类型冲突。解决方式是把 `RuntimeAbility` 改成交叉类型，并通过 `getRuleIndex` 在内部边界读取 hidden index。

4. `toRuntimeSubjectRequest` 的 value narrowing 问题

   初次 `yarn typecheck` 报错：

   ```txt
   Property 'value' does not exist on type '{ subject: string } | { value: object | TypedSubject<string, object> }'
   ```

   原因是 request 类型是 union，直接读取 `request.value` 不能通过窄化。解决方式是复用 `hasOwnValue(request)` guard 后再读取。

5. alias 反向不成立的运行时测试被类型系统提前拦住

   在 `tests/Ability.test.ts` 中直接用 `action: "modify"` 检查只有 `update/delete` 规则的 ability 时，类型检查报错：

   ```txt
   Type '"modify"' is not assignable to type 'ExpandedAction<"update" | "delete", ...>'
   ```

   这是预期的类型保护。为了测试运行时 one-way 语义，测试中将该 action cast 为 `never`，类型层反向不成立则由 `tests/Ability.type-tests.ts` 用 `@ts-expect-error` 覆盖。

6. `assert.throws` 不能接受 predicate matcher

   初版测试用 `(error: unknown) => boolean` 作为 `assert.throws` 第二参数，TypeScript 不接受。解决方式是改为传 `Ability.AliasError` 构造器。

7. `actionsFor` 测试期望漏掉 deny action

   第一次 `yarn test` 失败：

   ```txt
   expected [ 'access', 'delete', 'manage', ... ] to deeply equal [ 'access', 'manage', 'read', 'update' ]
   ```

   实际结果包含 `delete`，因为 `actionsFor` 表示“subject 上有规则的 action”，包括 deny 规则。测试期望调整为包含 `delete` 后通过。

8. `tsx` 示例运行遇到沙箱 IPC 错误

   普通沙箱运行：

   ```sh
   yarn tsx examples/basic.ts
   ```

   失败：

   ```txt
   Error: listen EPERM: operation not permitted /tmp/tsx-1000/38.pipe
   ```

   判断原因是 `tsx` 需要在 `/tmp` 下创建 IPC pipe，当前沙箱不允许。按仓库说明使用提升权限重跑 `examples/basic.ts`、`examples/advanced.ts`、`examples/acbc.ts`，三个示例均通过。
